# OA PRD 修改建议

## 文档目的

本文件是针对原《OA_PRD.md》的增量修改建议，重点解决原文缺少信息门户、流程中心、应用中心、通讯录、管理后台和企业信息的问题，并同步补齐应用中心中的固资管理入口与车辆管理模块定义。原 PRD 已对人资、费控和综合业务进行了大量页面与字段定义，本次以补齐全局入口、页面跳转和平台管理能力为主，车辆管理仅扩展到可支撑当前原型的必要范围。

## Codex 执行约束

### 输入材料

1. 原始业务文档：`OA_PRD.md`。
2. 现有系统原型代码。
3. 信息门户、流程中心、应用中心三张参考图片。
4. 本文件：`OA_PRD_修改建议.md`。

### 执行方式

1. 以原 `OA_PRD.md` 为业务基线，在原文档基础上增量修改，不删除原有业务模块。
2. 在现有原型代码基础上修改，不重新创建一套相互独立的原型。
3. 保持现有技术框架、组件库、页面风格、路由结构和模拟数据组织方式。
4. 优先复用已有页面、组件、数据对象和业务路由，避免生成重复页面、重复组件和重复数据源。
5. 三张参考图片用于确定页面布局、模块分区和视觉结构，不要求像素级照搬，但不得改造成完全不同的产品形态。
6. 不扩展本文件未明确要求的新业务模块；需要补齐页面细节时，应参照原 PRD 中相邻模块的写法和现有原型风格实现。
7. 本次不新增单独的验收标准章节。

### 输出内容

1. 修改后的完整 `OA_PRD.md`。
2. 修改后的系统原型代码。
3. 修改文件清单，说明新增、修改和删除的页面、组件、路由及数据结构。
4. 对受现有代码、依赖或数据限制而无法完成的内容单独说明，不得静默忽略。

## PRD 同步修改要求

新增或调整功能时，Codex 必须同步更新原 PRD 中的相关位置，不得只在文档末尾追加补充章节：

1. 背景与产品目标。
2. 用户与角色。
3. 本期范围和版本范围。
4. 术语定义。
5. 功能页面清单。
6. 数据对象与关键字段。
7. 权限设计。
8. 核心业务详写。
9. 系统功能和页面导航。
10. 与图片生成、二维码、H5 打卡相关的非功能说明。

具体同步要求：

- 信息门户、流程中心、应用中心、通讯录和管理后台纳入系统整体信息架构。
- 企业信息和用户管理补充到系统功能。
- 车辆管理补充到综合模块范围、功能页面清单和核心业务说明，详细程度按本文件第 9.2 节控制。
- 固资管理仅调整模块入口和命名，不删除原固定资产业务页面及业务对象。
- 不新增本次未要求的业务能力和验收标准。

## 二、全局导航与页面框架修改

### 2.1 顶部导航

所有普通业务页面固定保留：

```text
企业Logo 企业简称     信息门户｜流程中心｜应用中心       消息｜admin⌄
```

- Logo 和企业简称读取“管理后台 > 企业信息”。
- 消息展示未读数，点击打开消息列表或抽屉。
- admin 下拉菜单至少包含“管理后台、退出登录”。若现有系统已经具备个人中心则保留；若不存在，本次不新增个人中心。
- 管理后台仅对有权限用户显示。

### 2.2 页面形态

| 页面形态 | 左侧菜单 | 应用 |
| --- | --- | --- |
| 全局单页面 | 不展示系统级业务菜单，允许页面内部树/页签 | 通讯录、知识库 |
| 独立应用 | 仅展示本应用下级功能 | 会议管理、公文管理 |
| 业务工作区 | 展示业务域一级模块 | 组织文化、费控管理、人事管理、行政服务 |
| 全局工作区 | 展示全局能力自身菜单 | 流程中心 |
| 外部系统 | 不展示 OA 业务菜单 | ERP |

不要为了统一外观给通讯录、知识库等单页面增加空左侧菜单。

## 三、信息门户新增建议

### 3.1 页面布局

按门户原型补充以下卡片：

1. 门户横幅。
2. 快捷方式。
3. 待我处理、我已处理、我发起的、抄送我的。
4. 通知公告。
5. 业务入口 ERP。
6. 最新公文。
7. 最热知识。
8. 企业新闻。
9. 制度规范。

### 3.2 快捷方式

默认展示：通讯录、请假申请、费用报销、会议预约、用章申请、加班申请、补卡申请、办公用品、知识库、全部应用。

规则：

- 按用户权限过滤，无权限入口不占位。
- 申请类入口优先直接打开新增页/弹窗，减少二次跳转。
- “全部应用”进入应用中心。
- 门户快捷入口只作为跳转，不复制业务表单和数据。

### 3.3 流程卡片

- 每个页签默认 5 条。
- 展示流程标题、申请人、申请时间。
- 点击进入与流程中心相同的详情页。
- 查看更多进入流程中心对应菜单。
- 处理完成后局部刷新门户列表、统计和消息未读数。

### 3.4 内容卡片

| 卡片 | 数据来源 | 排序 |
| --- | --- | --- |
| 通知公告 | 通知公告 | 置顶优先，发布日期倒序 |
| 最新公文 | 公文管理 | 发布日期倒序 |
| 最热知识 | 知识库 | 热度倒序，发布日期作为次序 |
| 企业新闻 | 新闻管理 | 发布日期倒序 |
| 制度规范 | 制度中心 | 当前有效版本，发布日期倒序 |

仅展示已发布、在有效期内且用户位于可见范围的数据。

### 3.5 门户范围边界

- 本期按参考图片实现固定门户布局。
- 不新增门户装修、卡片拖拽、个人布局配置、复杂模板管理或多套门户切换。
- 门户卡片只复用现有业务数据和路由，不在门户内复制业务表单或单独建立业务详情页。

### 3.6 门户与流程中心名称映射

| 信息门户名称 | 流程中心名称 | 数据关系 |
| --- | --- | --- |
| 待我处理 | 我的待办 | 同一任务列表 |
| 我已处理 | 我的已办 | 同一处理记录 |
| 我发起的 | 我发起的 | 同一流程实例 |
| 抄送我的 | 抄送我的 | 同一抄送记录 |

## 四、流程中心新增建议

### 4.1 左侧菜单

```text
首页
发起流程
我的待办
我的已办
我发起的
抄送我的
```

### 4.2 首页组件

- 统计卡片：我的待办、我的已办、我发起的、抄送我的。
- 流程列表：四个页签，默认 5 条。
- 我的草稿。
- 常用流程。
- 全部流程：左侧分类树，右侧流程卡片。

### 4.3 分类命名

流程分类与应用中心统一：协同办公、组织文化、费控管理、人事管理、行政服务。将原型中的“日常办公”统一改为“协同办公”。

命名调整边界：

- 应用中心一级应用和人事管理工作区一级菜单统一命名为“组织架构”。
- 组织架构模块内部原有的“组织管理、组织架构图、岗位管理”等二级功能可以保留。
- 不对原 PRD 中“组织管理”二级页面进行无差别全文替换，避免形成“组织架构 / 组织架构”的重复层级。

### 4.4 数据复用

门户、消息和流程中心共用：流程实例 ID、任务 ID、业务单据 ID、详情路由。禁止生成独立的“门户待办详情”。

## 五、应用中心修改建议

### 5.1 分组

| 分组 | 应用 |
| --- | --- |
| 协同办公 | 通讯录、流程中心、会议管理、知识库、公文管理 |
| 组织文化 | 通知公告、制度规范、新闻管理 |
| 费控管理 | 预算管理、费用申请、费用报销、发票夹、费用类型 |
| 人事管理 | 组织架构、员工管理、入转调离、考勤管理、招聘管理、绩效管理、薪酬管理 |
| 行政服务 | 固资管理、办公用品管理、证照管理、用章管理、车辆管理 |
| 业务应用 | ERP |

### 5.2 布局

- 左侧分组用于定位右侧卡片。
- 分组标题列固定宽度，所有应用从同一条纵向起始线排列。
- 应用图标和名称按固定单元宽度排列，同列纵向对齐。
- 应用过多时整项换行，不缩小图标和文字。
- 无权限应用隐藏；分组无可见应用时整组隐藏。

### 5.3 跳转

| 应用 | 跳转结果 |
| --- | --- |
| 通讯录 | 独立单页面；页面内部组织树 + 人员列表 |
| 流程中心 | 切换顶部流程中心 |
| 会议管理 | 独立应用；展示会议自身菜单 |
| 知识库 | 独立单页面；页面内部分类树 |
| 公文管理 | 独立应用；发文登记/收文登记 |
| 组织文化三个应用 | 进入组织文化工作区并选中对应模块 |
| 费控五个应用 | 进入费控工作区并选中对应模块 |
| 人事七个应用 | 进入人事工作区并选中对应模块 |
| 行政服务应用 | 进入行政服务工作区并选中对应模块；左侧至少包含固资管理、办公用品管理、证照管理、用章管理、车辆管理 |
| ERP | 按现有配置打开独立系统；未配置时提示“ERP 系统尚未配置” |

## 六、通讯录新增建议

### 6.1 页面结构

通讯录使用“左侧组织树 + 右侧人员列表”，这里的左侧是页面内容，不是系统导航。

列表字段：头像、姓名、工号、部门、岗位、手机、邮箱、操作。默认只展示在职和试用期员工。

操作列至少包含：

- 查看详情
- 导出名片

数据要求：

- 复用组织和员工主数据，不建立第二套通讯录数据。
- 本期通讯录中的手机号码和邮箱直接读取员工主数据，不做字段脱敏。
- 不展示身份证、银行卡、薪资、绩效、家庭成员等与通讯录无关的敏感字段。

### 6.2 直接新增“导出名片”功能

Codex 必须在通讯录人员列表操作列和人员详情抽屉中增加“导出名片”入口，并实现以下功能，不得只制作静态按钮：

1. 点击后打开名片预览弹窗。
2. 弹窗根据最新企业信息、员工信息、部门和岗位生成预览。
3. 用户点击“导出 PNG”后下载单张 PNG 名片图片。
4. 本期仅导出 PNG，不提供 VCF 文件下载，不提供批量导出和模板设计器。
5. PNG 中必须包含联系人二维码；扫码后进入手机“添加联系人”界面。

### 6.3 名片内容

名片只展示以下内容：

- 企业 Logo
- 企业简称
- 员工姓名
- 部门
- 岗位
- 手机号码
- 邮箱
- 联系人二维码

其中企业 Logo 和企业简称读取“管理后台 > 企业信息”；其他字段读取员工、组织和岗位主数据。手机号或邮箱无值时自动隐藏。

### 6.4 联系人二维码

- 二维码使用标准 vCard 联系人文本编码，但不单独生成或下载 `.vcf` 文件。
- 二维码写入员工姓名、企业简称、部门、岗位、手机号码和邮箱。
- 扫码后由手机系统显示“新建联系人/添加到通讯录”，用户确认后保存。
- 二维码字段与 PNG 中展示的联系人字段保持一致。
- 二维码生成或校验失败时，不允许导出图片，提示用户重新生成。

### 6.5 PNG 与交互要求

| 项目 | Codex 实现要求 |
| --- | --- |
| 图片格式 | 仅 PNG。 |
| 建议尺寸 | 1200 × 720 px 横版。 |
| 文件名称 | `{员工姓名}_{企业简称}_名片.png`。 |
| 弹窗按钮 | 取消、导出 PNG。 |
| 字段为空 | 自动隐藏，不显示空标签。 |
| 导出范围 | 仅支持单人导出。 |
| 操作日志 | 记录操作人、被导出员工、导出字段、时间、IP 和结果。 |
| 异常处理 | 生成失败或员工数据失效时给出明确提示。 |

### 6.6 本期范围说明

- 本期不为“导出名片”单独增加按钮权限。
- 用户能够进入通讯录并查看员工信息时，即可导出该员工的 PNG 名片。
- PNG 和联系人二维码中的手机号码、邮箱直接使用员工主数据，不做脱敏处理。
- 本期仅支持单人导出，不提供批量导出。

### 6.7 通讯录数据范围特例

1. 通讯录是员工主数据的只读公共视图，不套用员工档案页面“普通员工仅查看本人”的数据范围。
2. 所有能够访问通讯录的用户，默认可查看在职和试用期员工。
3. 通讯录只开放头像、姓名、工号、部门、岗位、手机号码和邮箱，不展示员工档案中的其他敏感信息。
4. 通讯录中的手机号码和邮箱本期直接展示，不做脱敏。
5. 导出名片不单独增加权限；能够查看该员工通讯录信息即可导出。
6. 上述规则仅适用于通讯录和名片导出，不影响员工档案、薪资、绩效等模块的原有权限。

## 七、管理后台补充建议

### 7.1 入口和菜单

admin 下拉菜单增加“管理后台”。管理后台左侧菜单：

```text
企业信息
用户管理
角色管理
权限管理
流程管理
```

管理后台页面框架：

1. 管理后台使用独立的后台管理布局。
2. 左侧只展示企业信息、用户管理、角色管理、权限管理和流程管理。
3. 不展示人事、费控、行政服务等普通业务菜单。
4. 页面顶部提供“返回工作台”或“返回应用中心”入口。
5. 管理后台入口仅对具备管理权限的用户显示。

### 7.2 企业信息

字段只保留必要内容：企业全称、企业简称、企业 Logo。

- 企业简称和 Logo 用于全局左上角。
- 支持 Logo 预览、替换、恢复默认。
- 保存后刷新页面全局生效。
- 企业信息不替代组织架构中的公司主数据。

### 7.3 用户管理

列表字段：用户名、姓名、绑定员工、部门、手机号、角色、状态、最后登录时间。

操作：新增、编辑、启用、停用、重置密码、分配角色。

规则：员工离职后按策略停用账号；最后一个系统管理员不可停用；系统管理员不默认获得薪资等业务权限。

### 7.4 原系统功能迁移

原 PRD 中“流程管理、角色管理、权限管理”页面定义保留，但入口从普通业务菜单调整到管理后台。

## 八、考勤打卡边界修改

1. 删除应用中心“协同办公”分组中的考勤打卡入口。
2. 删除 OA Web 端考勤打卡页面、菜单、按钮和相关路由。
3. 原 PRD 功能页面清单中的“考勤打卡”调整为独立 H5 页面，不属于 Web 端人事管理菜单。
4. 全文检索并删除“电脑端打卡”“PC 打卡”等实现方式。
5. 员工打卡仅保留独立 H5 页面，支持地点打卡、Wi-Fi 打卡和外勤打卡。
6. 考勤机打卡可通过接口同步，不要求在本次原型中实现设备管理。
7. PC 端考勤管理继续保留考勤组、班次、排班、规则、原始打卡记录及日/月考勤统计。
8. H5 与 PC 考勤管理共用同一套员工、班次、规则和打卡记录数据。

原 PRD 中关于打卡方式的描述统一修改为：

> H5 支持地点、Wi-Fi 和外勤打卡；考勤机可通过接口同步。PC Web 不作为员工打卡方式。

## 九、范围对齐与补充建议

### 9.1 固资管理

原 PRD 已有完整固定资产模块，本次应在应用中心“行政服务”分组中补充“固资管理”入口，保持“功能有入口、入口有页面”。

命名和实现规则：

1. 应用中心卡片、行政服务一级菜单、模块标题和 PRD 模块名称统一使用“固资管理”。
2. 原 PRD 中作为模块名称使用的“固定资产管理”统一改为“固资管理”。

### 9.2 车辆管理

Codex 需要在现有 OA 原型和 PRD 中新增“车辆管理”功能，并将其放入“行政服务”工作区。点击应用中心中的“车辆管理”后，进入行政服务工作区并默认选中该模块。

车辆管理仅包含以下三个功能：

| 功能 | 大致说明 |
| --- | --- |
| 车辆信息 | 维护企业车辆的基础资料和当前状态，例如车牌号、车辆类型、所属部门、当前里程、年检和保险到期信息等。 |
| 用车申请 | 员工以本人驾驶为默认场景，选择可用车辆，填写用车时间、目的地和用车事由后提交审批。 |
| 用车归还 | 用车结束后登记实际归还时间、结束里程、油量和异常情况，并更新车辆状态。 |

实现要求：

1. 默认按员工本人驾驶车辆的场景设计，不建设司机管理，也不建设派车管理。
2. 用车申请中不设置“是否需要司机”“司机选择”等字段。
3. 同一车辆在时间重叠的有效用车申请中不可重复选择。
4. 停用、维修中或年检、保险不满足用车日期要求的车辆不可选择。
5. 车辆归还后更新车辆当前里程和可用状态；存在异常时可登记异常情况。
6. 不扩展车辆费用、保养计划、违章管理等其他功能。
7. 具体页面布局、查询条件、列表字段、表单字段、状态和交互细节，由 Codex 参照 PRD 中其他综合模块的书写和原型风格完成，避免超出上述范围。
8. 行政服务左侧“车辆管理”下只展示：车辆信息、用车申请、用车归还。
